type Effect = (Type -> Type) -> Type -> Type Source #

class EffectInject (op :: Effect) (ops :: [Effect]) | ops -> op Source #

Is the given type of effects an element of the list of effect types? Users won't write instances of this type, they're generated automatically.

In writing "reification" instances, a useful idiom is something like

instance EffectInject (WriterEffect w) ops => MonadWriter (AST ops) where
  ...

which will define "writer-like syntax" for all lists of effects that contain 'WriterEffect w'. This idiom is also the reason for the functional dependency in the class: ops must contain enough information to determine op.

If all effect types in the AST have an 'InterpretEffect m' instance, this function will interpret the AST.

class InterpretEffectStateful t m op where Source #

Users will write an instance of this class if they want to interpret effects of type op in m, while also passing a state of type x form step to step. This state will control how effects are implemented.

For example in Fixpoint.Examples, I play around with the type of LTL formulas for x.
